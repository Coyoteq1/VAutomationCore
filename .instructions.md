# VAutomationCore Development Guidelines

VAutomationCore is a **reusable NuGet library** for V Rising server mods. It provides shared ECS helpers, lifecycle management, event infrastructure, zone systems, and configuration utilities consumed by companion plugins (e.g., Blueluck).

> General V Rising modding patterns (ECS, Harmony, prefabs) are in the shared `vrising-modding.instructions.md`. This file covers VAutomationCore-specific conventions only.

## Project Identity

- **Package**: `VAutomationCore` on NuGet (v1.1.1)
- **GUID**: `gg.coyote.VAutomationCore`
- **Target**: .NET 6.0, `<Nullable>annotations</Nullable>` (NOT `enable`), unsafe allowed
- **Output**: Library DLL + NuGet package (`.nupkg` / `.snupkg`)
- **Build output**: `artifacts/nuget/`
- **Deterministic builds**: enabled (SourceLink, embedded sources, portable debug symbols)
- **XML docs**: generated via `<GenerateDocumentationFile>true</GenerateDocumentationFile>` (warns suppressed: 1591)
- **Assembly references**: conditional `UseServerAssembliesForBuild` for server interop DLLs

- `VAuto.Core` — PrefabGuidExtensions

- `VAutomationCore.Extensions` — ECSExtensions (file namespace is `VAuto.Extensions`)

## Architecture

### UnifiedCore — Static World Accessor (`VAutomationCore.Core`)
- Static class exposing `Server` (World), `EntityManager`, `PrefabCollection`.
- **Lazy initialization** — first access resolves via `GetWorld("Server")` from `World.s_AllWorlds`.
- `IsInitialized` guard must be checked before accessing game systems in early patches.
- `Reset()` clears cached state.
- **IMPORTANT**: `VAutoExtensions.cs` references `VRCore.EntityManager` — this is a naming bug. The canonical name is `UnifiedCore`. `ECSExtensions.cs` correctly uses `UnifiedCore.EntityManager`.

### IService Hierarchy (`VAuto.Services.Interfaces`)
Three-tier interface contract defined in `IService.cs`:

| Interface | Adds | Use Case |
|-----------|------|----------|
| `IService` | `IsInitialized`, `Log`, `Initialize()`, `Cleanup()` | Base for all services |
| `IEntityService : IService` | `RegisterEntity()`, `UnregisterEntity()`, `GetEntityCount()` | Services managing tracked entities |
| `IArenaService : IEntityService` | `CreateArena()`, `DeleteArena()`, `GetArenaIds()` | Arena lifecycle management |

- Services are instantiated during initialization, not in constructors.
- **Note**: Blueluck services do NOT implement these interfaces — they are plain classes. These interfaces are for VAutomationCore's own services.

### VAutoPatch — Abstract Base Class (`VAutomationCore.Core.Patches`)
Defines reusable guard patterns as `protected static` methods:
- `IsServerReady` — delegates to `ServerReadySystem.Instance.IsServerReady`.
- `HasEventSubscribers<T>()` — checks TypedEventBus before allocating arrays.
- `EmitEvent<T>(eventData)` — publish to TypedEventBus with subscriber check.
- `EmitEntitiesEvent<T>(entities, factory)` — bulk-emit per entity.

**In practice**: No patches currently inherit from VAutoPatch. `VAutoPatches.cs` calls `ServerReadySystem.Instance.IsServerReady` and `TypedEventBus.Publish()` directly. VAutoPatch exists as a convenience base class for future use.

### TypedEventBus (`VAutomationCore.Core.Events`)
Static publish-subscribe system. Thread-safe via `lock(Sync)`.
- `TypedEventBus.Subscribe<T>(handler)` / `Unsubscribe<T>(handler)`.
- `TypedEventBus.SubscribeScoped<T>(handler)` — returns `IDisposable` for auto-unsubscribe.
- `TypedEventBus.Publish<T>(eventData)` to emit.
- `TypedEventBus.HasSubscribers<T>()` — check before expensive event construction.
- Internal storage: `Dictionary<Type, List<Delegate>>`.

### ServerReadySystem (`VAutomationCore.Core.Services`)
Singleton managing server lifecycle state:
- `ServerReadySystem.Instance` — lazy singleton.
- `SetServerReady()` — marks ready, executes queued `_onReadyCallbacks`.
- `SetShardTaken()` — marks shard taken, notifies via `ModAPI.Broadcast`.
- `OnReady(callback)` — register callback; fires immediately if already ready.
- `IsServerReady` property.

### Extension Methods (`Extensions/`)

| File | Namespace | Accessor | Pattern |
|------|-----------|----------|---------|
| `VAutoExtensions.cs` | `VAuto.Extensions` | `VRCore.EntityManager` (bug — should be `UnifiedCore`) | `GetOrAdd<T>`, `GetSafe<T>`, `GetOrDefault<T>`, `SetIfExists<T>`, `HasAllComponents` |
| `ECSExtensions.cs` | `VAuto.Extensions` | `UnifiedCore.EntityManager` | `Read<T>`, `Write<T>`, `Has<T>`, `Add<T>` via Marshal/byte marshaling |
| `PrefabGuidExtensions.cs` | `VAuto.Core` | — | `LookupName()` on PrefabGUID |

### ChatService (`VAuto.Core.Chat`)
Static wrapper around game messaging via `GameActionService`:
- `TrySendSystemMessage(platformId, message, out error)` — individual.
- `TryBroadcastSystemMessage(message, out error)` — server-wide.
- Messages trimmed via `TrimForFixedString()` for FixedString compatibility.
- Returns `bool` + `out string error` (never throws).

### ZoneLifecyclePatch — Delegate Injection
Uses delegate fields for composable zone behavior:
- `NormalizeZoneId` — canonical zone identifier mapping.
- `OnBeforeEnter` / `OnAfterEnter` — pre/post zone entry hooks.
- `OnBeforeExit` / `OnAfterExit` — pre/post zone exit hooks.
- Consuming mods subscribe to zone events via these delegates or `TypedEventBus`.

## Configuration — Three Layers

| Layer | Class | Purpose |
|-------|-------|---------|
| Static metadata | `PluginManifest` (`VAuto.Core.Configuration`) | Plugin key, GUID, name, version, Harmony toggle |
| Runtime settings | BepInEx `ConfigFile` binding in `Plugin.Load()` | Hot-reloadable via `LifecycleConfigWatcher` |
| JSON data | `TypedJsonConfigManager<T>` | Zone presets, flows, etc. Custom `JsonConverter` support (PrefabGUID, enums) |

- Config files live in `BepInEx/config/` subdirectories.
- Hot-reload: file watcher triggers `Reload()`.

## Logging

- `Plugin.CoreLog` — `ManualLogSource` for BepInEx-level logging.
- `CoreLogger` (`VAutomationCore.Core.Logging`) — structured logging with `[CallerMemberName]` context.
- `CoreLogger.Initialize()` must be called during plugin load before any logging.
- Pattern: `CoreLogger.Info($"[Context] message")`.

## GlobalUsings.cs

Uses regular `using` (NOT `global using`):
- `VAutomationCore.Core`, `VAutomationCore.Core.Logging`, `VAutomationCore.Core.Services`
- `VAutomationCore.Core.ECS`, `VAutomationCore.Core.Config`
- `VAutomationCore.Patches`, `VAutomationCore.Extensions`
- `VAutomationCore.Json`, `VAutomationCore.Configuration`, `VAutomationCore.Models`
- `Stunlock.Core`

## Models

- `BloodType` — PrefabGUID-based vampire blood type tracking.
- `GlowZoneEntry` — Visual glow effect per zone (center, radius, color).
- `PlayerZoneState` — Zone transition state machine with debouncing.
- `TechPrefab` — Technology/progression unlock data.

### ECS Components (`Core/ECS/Components/`)
- `EcsPlayerZoneState` — current zone hash.
- `ZoneComponent` — zone metadata tag.
- `ZoneTransitionEvent` — lifecycle hook event data.

### Data Files (`Data/`)
- `Prefabs.cs` — PrefabGUID named constants.
- `Territory.cs` — territory definitions.
- `VBloods.cs` — V Blood boss data.
- `WorldRegions.cs` — world region mappings.
- `buffs_numbered.txt`, `vbloods_bosses.json` — embedded data files.

## NuGet Packaging Conventions

- `GeneratePackageOnBuild = true` — every build produces a package.
- SourceLink enabled for source debugging.
- Symbols in `.snupkg` format with `EmbedUntrackedSources`.
- `PackageOutputPath = artifacts/nuget/`.
- Assembly references: conditional `UseServerAssembliesForBuild` for switching between NuGet refs and local server interop DLLs.
- All public API surface must have XML doc comments.
